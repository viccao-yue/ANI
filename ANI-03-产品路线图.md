> 版本 V1 | 广州常青云科技有限公司 | 内部产品规划文件

## 一、路线图设计原则

基于公司现状（品牌积累有限、产品处于起步阶段、销售能力聚焦非开发场景），路线图遵循以下逻辑：

1. **硬性截止：2026-09-30 交付第一版本**，倒推所有里程碑
2. **AI 开发为主，人工为辅**：接口设计和架构决策由人主导，代码实现最大化由 AI 工具生成
3. **复用优先**：算力层充分复用公司现有虚拟化/超融合/容器能力，不重新造轮子
4. **从产品最薄处开始**：知识库问答和模型私有化部署是门槛最低、客户最能感知价值的场景，先打通
5. **用案例驱动信任**：每一阶段以"打造一个标杆客户"为核心目标，而非铺量

> **GPT 审查 P1 修复（2026-05）：Phase 1 范围大幅缩减**
> 原计划 Phase 1 同时包含 Karmada 多集群、Patch 升级 Operator、完整 BOSS、Agent 平台等重型功能，
> 与 5 个月 + AI 开发模式的实际产能严重不匹配。
> 按 GPT 建议，Phase 1 聚焦"能让国央企/高校客户签 POC 合同"的最小功能集，删除特性见下方。

## 二、三阶段路线图

### Phase 1：基础跑通（0–6 个月）

**目标：** 能签第一单、能完成交付、能有一个客户在用

**产品交付范围：**

| 模块 | 功能 | 优先级 |
|---|---|---|
| GPU 算力管理 | GPU 纳管、利用率监控、资源配额 | P0 |
| GPU 算力管理 | GPU 纳管、利用率监控、资源配额（HAMi + Volcano）| P0 |
| 模型管理平台 | 模型导入（离线包 + HuggingFace/ModelScope）、一键部署、动态加载 | P0 |
| 模型加解密 | 国密 SM4 对称加密，推理时 Init Container 内存解密（Secret volume 非 env）| P0 |
| 企业知识库问答 | 文档上传解析、RAG 问答（混合检索）、来源引用、幂等解析 | P0 |
| 基础安全管控 | RBAC 权限、AD/LDAP 集成、审计日志、RLS fail-closed | P0 |
| ani-installer | 裸机/VM/已有K8s 三种模式，硬件检测，GPU驱动安装，内部CA | P0 |
| Console 前端（核心页面）| 仪表盘、模型管理、知识库管理、用量报表 | P0 |
| BOSS 前端（简化版）| 租户管理、资源配额、平台健康大盘 | P1 |
| Harbor 镜像仓库管理 | Console 封装 Harbor API（镜像浏览）| P1 |
| CLI 工具（ani）| 核心命令：model / kb / cluster | P1 |
| 可观测性基础 | Prometheus + Grafana（GPU + 推理大盘）| P1 |

**从 Phase 1 删除的功能（推至 Phase 2，GPT 审查建议）：**
- ~~Karmada 多集群 Region/AZ~~ → Phase 2（单集群先跑通）
- ~~Patch 升级 Operator + BOSS 升级界面~~ → Phase 2
- ~~BOSS 白牌化（Logo/ICO/主题色）~~ → Phase 2
- ~~Agent 平台（AgentToolPermission / 沙箱）~~ → Phase 2
- ~~等保三级合规完整文档~~ → Phase 2
- ~~信创适配（UOS + ARM64 构建）~~ → Phase 2（时间不够时延后）

**不做（推至 Phase 2）：** 文档智能处理、会议智能、视频智能、微调平台

**目标客户：** 1–2 个国央企或大型高校（优先有 GPU 服务器但利用率低的客户）

**成功标准：**
- 完成一次从部署到验收的完整交付
- 知识库问答累计响应 1000 次以上（证明业务用户在实际使用）
- 获得可对外引用的客户证言

### Phase 2：场景深化（6–12 个月）

**目标：** 形成 2–3 个可复制的行业解决方案包，拓展至金融或医疗行业

**产品交付范围：**

| 模块 | 功能 | 优先级 |
|---|---|---|
| 文档智能处理 | 合同要素提取、公文起草、文档摘要 | P0 |
| 会议智能 | 语音转写、发言人区分、纪要生成 | P0 |
| 模型微调平台（轻量版）| 数据标注、LoRA 微调、效果评测 | P1 |
| 视频内容智能（基础版）| 视频内容检索、字幕生成 | P1 |
| 企微/钉钉 Bot 集成 | 知识库问答 Bot | P1 |
| 等保三级合规架构 | 合规架构设计 + 配置文档 | P0 |
| 运维控制台（完整版）| 应用大盘、模型大盘、灰度升级 | P1 |

**重点行业解决方案包：**

- **金融行业包**：合同审查 + 合规文档生成 + 内部制度知识库，附等保三级说明文件
- **国央企行业包**：公文智能起草 + 会议纪要 + 内部知识库，附信创部署说明
- **教育行业包**：教学资料知识库 + 科研文献问答 + 行政文档处理

**目标客户：** 在 Phase 1 基础上新签 3–5 个客户，覆盖至少 2 个行业

**成功标准：**
- 形成至少 1 个金融或医疗标杆客户（品牌背书价值高）
- 交付周期缩短至 4 周以内（标准化程度提升）
- 客户续费/扩容率 > 50%

---

### Phase 3：生态与开发场景（12–24 个月）

**前提条件：** Phase 2 完成后，产品力和品牌力已有一定积累，具备开拓开发场景的基础

**目标：** 在稳固非开发场景份额的同时，逐步打开面向开发者的市场

**产品交付范围：**

| 模块 | 功能 |
|---|---|
| AI 应用市场 | 第三方应用商入驻，客户按需订阅 |
| 开放 API 平台 | Prompt 管理、调用统计、计费 |
| Agent 编排框架 | 多步骤任务 Agent，面向有技术能力的客户 |
| 安全沙箱执行环境 | Agent 代码执行隔离（符合 ANI 产品定义中的 AI 原生能力） |
| 行业大模型微调服务 | 深度定制（全量微调，而非 Phase 2 的 LoRA 轻量版） |
| 多租户 SaaS 版本 | 面向集成商分发，降低渠道销售门槛 |

## 三、关键里程碑

```
Month 1–2   内部测试版本就绪（知识库问答 + 模型部署）
Month 3     拿到第一个 POC 客户
Month 4–5   完成第一次完整交付（包括验收文档）
Month 6     Phase 1 总结：产品缺陷修复 + 交付流程标准化

Month 7–8   文档智能 + 会议智能模块上线
Month 9     进入第一个金融行业客户 POC
Month 10–11 金融标杆客户合同签署与交付
Month 12    Phase 2 总结：行业解决方案包定稿，对外发布

Month 13–18 逐步开放 API 平台，试点开发者客户
Month 19–24 Agent 编排能力 + 多租户版本，开拓渠道合作
```

## 四、资源优先级建议

在产品人力有限的前提下，建议按以下优先级分配开发资源：

| 优先级 | 投入方向 | 原因 |
|---|---|---|
| 最高 | 知识库问答的可靠性和准确率 | 这是所有场景的基础，也是客户使用频率最高的功能，质量决定口碑 |
| 高 | 部署简化（安装包+部署工具）| 非标准化部署是当前最大的交付成本，每降低一天人天就是利润 |
| 高 | 信创兼容 | 国央企和金融的入场券，没有信创支持直接失去大量机会 |
| 中 | 文档处理准确率（OCR+表格）| 金融客户最高频场景，准确率不达标直接影响续费 |
| 中 | 安全审计功能完整性 | 等保验收必需，不做就无法通过甲方 IT 评审 |
| 低 | 界面美观度 | 企业客户对 UI 要求不高，功能正确比界面好看重要 |
| 最低 | 开放 API / 开发者工具 | Phase 3 再做，现阶段投入产出比极低 |

## 五、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|---|---|---|---|
| 大模型准确率不满足客户预期 | 高 | 高 | Phase 1 选择容错度高的场景（知识库问答），提前设置客户预期，配备人工兜底 |
| 信创适配耗时超预期 | 中 | 高 | Phase 1 若时间紧张，优先适配一种 OS（统信 UOS），其余延至 Phase 2 |
| GPU 服务器兼容性问题 | 中 | 中 | 建立硬件兼容性矩阵，签单前提前做硬件调研 |
| 竞品低价竞争（集成商） | 高 | 中 | 用标准化产品和更快的交付周期体现价值，而非打价格战 |
| 关键技术人员流失 | 低 | 高 | 核心能力文档化，减少对个人经验的依赖 |
| 客户采购流程拖延（超过6个月）| 高 | 中 | Phase 1 首选决策链短的客户（如高校），金融/国央企列入 Phase 2 |
