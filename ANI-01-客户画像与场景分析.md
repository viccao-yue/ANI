## 一、第一性原理推导：客户真正在买什么

从最基本的事实出发问三个问题：

**Q1：这些客户为什么不用公有云 AI？**
不是不知道，是不敢用。金融、国央企、医疗有明确的数据主权和合规要求——数据不能出内网，模型调用记录不能留在第三方服务器。这是根本约束，不是偏好。

**Q2：他们内部有没有 AI 工程能力？**
基本没有。IT 部门会运维服务器，但没有会训练模型、写 Prompt 工程的人。所以他们需要的不是"能力平台"，而是"开箱即用的 AI 应用"。

**Q3：他们的 AI 预算从哪里来？**
项目制拨款，不是订阅制。这意味着：要有立项理由（上级要求转型/同行已在用）、要有明确交付物、要能在年底前花完预算。产品必须适配"项目交付"逻辑，而非 SaaS 逻辑。

**推导结论：** 这类客户买的是"让现有工作用 AI 做得更快"，而不是"用 AI 开发新能力"。产品设计必须围绕**降低存量工作的人力成本**展开，而非围绕开发者体验。

## 二、客户画像

### 共性特征（跨行业）

| 维度 | 描述 |
|---|---|
| 数据要求 | 不出内网，物理隔离，等保合规 |
| IT 能力 | 能运维服务器，无 AI 研发能力 |
| 采购方式 | 项目制，一次性或年框，非订阅 |
| 决策链 | 长，需 IT 部门 + 业务部门 + 领导三方协同 |
| 硬件现状 | 多数已采购 GPU 服务器，利用率低 |
| 交付预期 | 交钥匙，供应商负责到底 |

### 行业细分

**金融（银行、保险、证券）**
- 预算规模最大，合规要求最严
- IT 部门有一定技术能力，但 AI 应用开发经验为零
- 核心痛点：每天海量合同、报告、客户文件需要人工审核，效率瓶颈显著
- 高价值场景：合规文本审查、监管报告生成、智能客服知识库、风险文档提取
- 关键门槛：金融信创（兼容国产芯片/OS）、等保三级、数据不出行

**国央企（能源、通信、制造类央企及政府机关）**
- 行政管理类 AI 需求为主，业务驱动力来自上级"数字化转型"政策要求
- 核心痛点：大量重复性行政工作（公文拟写、会议纪要、材料汇编）耗费人力
- 高价值场景：公文智能起草与审核、会议全程记录与摘要、内部知识库问答
- 关键门槛：信创优先、党政专网部署、国产化率要求

**教育（双一流高校、职业院校集团）**
- 预算相对有限，但采购周期明确（年初立项，年底验收）
- 核心痛点：教师备课繁琐、科研文献检索耗时、行政文档处理低效
- 高价值场景：教学资料智能生成、科研文献问答、学生答疑知识库
- 关键门槛：教育信创、校园网隔离部署

**医疗（三甲医院、医联体）**
- 预算中等，数据合规要求高（医疗数据分级保护）
- 核心痛点：医生书写病历和医疗文书占用大量诊疗时间
- 高价值场景：病历结构化录入辅助、医疗文书生成、院内知识库（规范/指南/药典）
- 关键门槛：医疗数据合规、HIS/EMR 系统对接、HIPAA 类国内等价要求

## 三、"非开发场景"的本质界定

用户明确指出，当前阶段聚焦**非开发场景**。从第一性原理界定其边界：

**非开发场景的核心特征：**
- 最终用户是业务人员，不是工程师
- 输入是已有数据（文档、视频、语音），输出是加工后的结果（摘要、回答、结构化数据）
- 不需要写代码，不需要调 API，不需要懂模型参数
- ROI 用"节省多少人时"来衡量，而非"能跑多少推理请求"

**当前值得进入的非开发场景（按优先级排序）：**

| 优先级 | 场景 | 核心价值 | 主要行业 |
|---|---|---|---|
| P0 | 企业知识库问答 | 替代人工查规程/找文件 | 全行业 |
| P0 | 模型私有化部署推理 | GPU 资产盘活，统一 API | 全行业 |
| P1 | 文档智能处理 | 合同/公文审核提速 | 金融、国央企 |
| P1 | 会议智能（转写+摘要） | 节省会议记录人力 | 国央企、教育 |
| P2 | 视频内容智能 | 监控/会议视频可检索 | 全行业 |
| P2 | 模型微调（行业专属） | 提升垂直领域准确率 | 金融、医疗 |
| P3 | OA/ERP AI 集成 | 流程自动化 | 国央企 |

**暂不进入的开发场景（留待 Phase 3）：**
- 面向开发者的 API 平台（Prompt 管理、SDK）
- AI 应用开发框架
- 业务流程 RPA + AI 联合开发

## 四、购买决策链分析

理解谁拍板、谁否决，是产品能否卖出去的关键。

```
业务部门提需求（感受痛点）
        ↓
IT 部门评估可行性（技术门槛、安全合规）
        ↓
采购/财务部门立项（预算合规、招标要求）
        ↓
分管领导审批（政治正确性，是否符合数字化战略）
```

**各角色关注点：**

| 角色 | 关注点 | 产品要给的答案 |
|---|---|---|
| 业务部门负责人 | 能解决我的什么具体问题？ | 场景案例 + 效果数据 |
| IT 部门 | 能和现有系统对接吗？安全吗？运维复杂吗？ | 架构图 + 安全白皮书 + 运维手册 |
| 采购/财务 | 价格是否符合采购规则？能开发票吗？ | 标准报价 + 资质证书 |
| 分管领导 | 别人在用吗？出问题谁负责？ | 标杆案例 + 服务承诺 |

**结论：** 产品材料必须准备多套，针对不同角色。技术文档给 IT，场景演示给业务，资质证书给采购，行业客户案例给领导。

## 五、竞争格局与切入策略

**主要竞品：**
- 华为云 MetaStudio、私有化部署方案（品牌强，但重，价格高）
- 百度智能云千帆（模型丰富，但倾向公有云）
- ModelScope/魔搭（开源生态，适合开发者，不适合非开发场景客户）
- 各地集成商（无核心产品，靠关系，质量参差）

**我们的差异化切入点：**

1. **比大厂更灵活**：华为、百度的私有化方案重，部署周期长，定制成本高。我们可以做到更快交付、更低门槛。
2. **比集成商更有产品**：本地集成商靠关系，没有标准化产品，维护能力弱。我们有自己的产品底座，售后可持续。
3. **场景预置深度**：针对目标行业预置好场景模板，客户不用自己配置，降低采购风险感知。

**不要试图在全行业同时铺开**，第一阶段选 1-2 个行业打透，形成标杆案例后再扩展。建议优先选择**国央企或大型高校**（预算合规、数据敏感度略低于金融、决策周期相对可控）。
