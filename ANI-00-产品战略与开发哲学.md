## 一、为什么要做这个产品

公司已有服务器虚拟化、超融合、私有化云平台、分布式存储、容器平台等产品线，但这些产品在市场上缺乏功能亮点和差异化竞争力，难以在同质化竞争中胜出。

**KuberCloud ANI（AI专有云）是公司突破现有产品格局的战略级产品。**

它完全从零开始构建，不继承现有产品的任何代码和架构约束，不受历史技术债务的限制。这是一次机会，也是一次要求——做出来的东西必须是生产级的平台，不能是走走形式的玩具。

## 二、开发哲学：开源编排，而非重复发明

### 2.1 核心思想

AI 专有云要做的事，不是重新发明 Kubernetes、重新写一个向量数据库、重新实现一个 LLM 推理引擎。这些领域已经有世界级的开源实现：Kubernetes、vLLM、Milvus、KubeOVN、HAMi……它们背后是数百名工程师多年的积累，不是我们应该去竞争的地方。

**我们的价值在于：**

```
选择正确的开源组件
    +
用正确的方式把它们组合在一起（编排）
    +
在它们之上构建面向用户的完整工作流（封装）
    +
让企业客户在不懂 AI 的情况下也能用起来
    =
伟大的产品
```

### 2.2 "编排"意味着什么

编排不是简单的 API 透传，而是：

- **统一控制平面**：Kubernetes + Volcano + HAMi + KubeOVN 对用户是四个独立系统，ANI 将它们统一为一个"GPU 资源管理"体验，用户看不到这四个系统的存在
- **工作流自动化**：从"上传模型文件"到"推理服务上线可调用"，中间涉及模型格式转换、配置生成、K8s Deployment 创建、服务注册、健康检测……ANI 将这些步骤自动串联，用户只做第一步和最后一步
- **状态收敛**：底层组件各有其状态，ANI 负责将这些状态收敛为一个用户可理解的单一状态模型（部署中 / 运行中 / 失败）
- **错误处理的人性化**：底层报错是 `OOMKilled` 或 `CrashLoopBackOff`，ANI 将其翻译为"内存不足，建议将并发数从 8 调整为 4"

### 2.3 "封装"意味着什么

封装是在编排之上，为具体业务场景构建完整的用户工作流：

- 知识库问答不只是调用 RAG 接口，而是从"上传文档→解析→向量化→建立知识库→配置问答策略→测试→上线"的完整闭环，每一步都有界面引导
- 模型微调不只是执行训练脚本，而是"准备数据→标注→配置微调参数→执行→评估对比→一键替换生产版本"的完整闭环
- 这些封装对应的是客户真实的工作场景，不是技术功能的简单罗列

### 2.4 我们要做的，类比

> macOS 底层是 BSD Unix，但没有人把 macOS 叫做"Unix 的封装"。  
> iPhone 的芯片、传感器都来自各种供应商，但没有人把 iPhone 叫做"供应链的组装"。

开源组件是原材料，ANI 的产品设计、工作流设计、用户体验、生产级可靠性，才是我们的核心竞争力。

## 三、AI 大模型辅助快速开发

### 3.1 基本立场

ANI 的开发从第一天就利用 AI 编码工具（Claude Code、Cursor 等）加速开发。这不是实验性尝试，而是核心开发策略。

**AI 辅助开发不是让 AI 替代工程师，而是让工程师的产出乘以 3-5 倍。**

### 3.2 为 AI 编码优化的架构设计

以下架构决策同时服务于"生产级质量"和"AI 辅助开发效率"两个目标：

**Spec-First 开发（OpenAPI / Protobuf 先行）**
- 接口定义写清楚，AI 可以直接根据 Spec 生成 Server Stub、Client、测试用例、Mock
- 不允许"先写代码，再补文档"的工作方式，因为这样 AI 辅助效率最低

**单一职责的小型服务**
- 每个微服务做一件事，上下文边界清晰
- AI 在边界清晰的上下文内生成的代码质量远高于在大型模糊系统中的表现

**强类型 + 明确的错误类型**
- Go 的强类型系统让 AI 生成的代码更容易编译通过
- 定义明确的 error type 而非模糊的 `error`，AI 能理解意图并正确处理错误路径

**测试先行**
- 先写接口定义和测试用例，再让 AI 生成实现
- 测试即规格，AI 以测试为目标生成的代码质量更可控

### 3.3 人机协作分工

| 任务类型 | 以人为主 | 以 AI 为主 |
|---|---|---|
| 架构决策与取舍 | ✓ | |
| 接口定义（OpenAPI/Proto）| ✓ | |
| 业务逻辑设计 | ✓ | |
| Server/Client 样板代码生成 | | ✓ |
| 单元测试和集成测试生成 | | ✓ |
| 错误处理代码 | | ✓ |
| 文档字符串和注释 | | ✓ |
| 重复性 CRUD 实现 | | ✓ |
| 代码审查与安全检查 | ✓ | ✓（辅助）|
| 性能优化决策 | ✓ | |

### 3.4 对 AI 辅助开发不友好的做法（禁止）

- 用魔法数字和隐式约定代替明确的类型定义
- 函数超过 100 行（AI 在长函数中容易丢失上下文）
- 在注释里写"TODO: 以后再改"（AI 会把注释当真）
- 接口文档和代码不同步（AI 会被误导）

## 四、生产级的具体定义

"不是玩具"必须有可量化的标准，不能靠感觉。

### 4.1 稳定性

| 标准 | 要求 |
|---|---|
| API 可用性 | 月度 99.9%（非计划内停机 ≤ 43 分钟/月） |
| 数据持久性 | 知识库、模型元数据、审计日志零丢失 |
| 故障自愈 | Pod 崩溃后 60 秒内自动重启并恢复服务 |
| 灰度升级 | 新版本先切 10% 流量，无错误后再全量，不允许停机升级 |
| 回滚能力 | 任何发布版本可在 5 分钟内回滚到上一个版本 |

### 4.2 可扩展性

| 标准 | 要求 |
|---|---|
| 水平扩展 | Web Server 和微服务层无状态，可无限水平扩展 |
| 多租户 | 单集群支持 100 个租户，租户间资源和数据完全隔离 |
| GPU 扩容 | 新增 GPU 节点后，10 分钟内纳入调度池，无需重启平台 |
| 存量兼容 | API 大版本升级保证 N-1 版本向后兼容 6 个月 |

### 4.3 可观测性（不可观测的系统不是生产级系统）

| 要求 | 实现 |
|---|---|
| 每个 API 请求有唯一 request_id，全链路可追踪 | OpenTelemetry Trace |
| 关键业务指标实时大盘（GPU 利用率、推理 QPS、错误率） | Prometheus + Grafana |
| 异常自动告警，分级（P0 立即响应 / P1 工作时间响应）| AlertManager |
| 审计日志完整记录，不可篡改 | 独立审计服务 + 追加写入 |

### 4.4 安全性

| 要求 | 标准 |
|---|---|
| 所有外部 API 强制 HTTPS（TLS 1.3） | 无论内网还是外网 |
| 认证 token 有过期时间，支持吊销 | JWT + 黑名单机制 |
| 敏感配置（密钥、密码）不出现在代码和日志中 | K8s Secret + Vault |
| 依赖组件定期扫描 CVE | CI 流程集成 Trivy 扫描 |
| 等保 2.0 三级合规架构 | 交付物包含合规说明文档 |

### 4.5 运维友好性

| 要求 | 标准 |
|---|---|
| 部署 | Helm Chart 一键部署，首次部署 < 2 小时 |
| 升级 | 平滑升级，不中断现有推理服务 |
| 备份 | 关键数据（知识库、配置）定时自动备份，可一键恢复 |
| 断网运行 | 所有功能在完全断网环境下正常运行，不依赖任何外部服务 |
| 离线安装包 | 提供完整的离线安装包（镜像 + Helm Chart + 安装脚本） |

## 五、"伟大产品"的评判标准

产品开发过程中，遇到任何功能设计、技术决策时，用以下问题检验：

**对用户：**
- 这个功能，一个不懂 AI 技术的 IT 管理员，能在 30 分钟内学会用吗？
- 出错时的提示，能让用户知道"下一步该怎么办"吗？
- 如果这个功能删掉，客户会打电话投诉吗？（不会的，大概率不该做）

**对系统：**
- 如果这个服务挂掉，整个平台会跟着挂掉吗？（不应该）
- 如果客户量增长 10 倍，这个设计还能撑住吗？
- 3 年后，今天写的接口还能继续用吗？

**对团队：**
- 一个新加入的工程师，能在 1 天内理解这部分代码并做出修改吗？
- 这部分逻辑，能被 AI 辅助工具高效完成吗？如果不能，是不是设计得太隐晦了？
