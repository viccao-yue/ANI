# PRD: Console 对象存储

## 1. 页面定位

`Console / 算力与云资源 / 存储管理 / 对象存储` 用于帮助租户用户查看、创建、删除和理解当前权限范围内的对象元数据资源，并为 `VM`、容器、`GPU 容器` 等工作负载提供统一的对象资源入口。

本页属于 `Console` 租户侧资源管理页，不是 `BOSS` 的平台对象存储池运营页。

## 2. 权威源结论

当前一级权威源以 `ANI-main/repo/api/openapi/v1.yaml` 为准，可确认：

- `StorageObject` 属于 `Core`
- 当前已冻结的对象存储路径仅包括：
  - `/api/v1/objects`
  - `/api/v1/objects/{object_id}`
- 当前尚未冻结：
  - 对象上传
  - 对象下载
  - 桶资源
  - 桶策略

## 3. Goals

- 让租户在对象存储页面内理解当前租户下对象元数据资源的边界、状态和基础能力
- 严格以现有 `Core v1.yaml` 为准，沉淀对象存储页面可直接用于 `Core` 对齐的主维护材料
- 清楚区分哪些对象存储能力已经冻结，哪些仍为待补能力
- 为 `VM`、容器、`GPU 容器` 等页面提供统一的对象资源回指入口
- 保持 `Console` 租户视角，不混入平台对象存储池运营语义

## 4. 用户故事

### US-001: 查看对象存储总览

作为租户用户，我希望在对象存储页面中快速理解当前对象资源范围和边界，以便确认哪些能力已经可用。

验收标准：

- 页面明确这是 `Console` 租户侧的对象存储资源页
- 页面明确当前只承接对象元数据列表、详情、创建、删除
- 页面明确标出对象上传、对象下载、桶策略、桶资源仍为待补能力
- 页面明确说明当前不是完整对象内容传输页
- 页面不混入平台侧全局对象存储池语义

### US-002: 查看对象元数据列表和详情

作为租户用户，我希望查看对象元数据列表和详情，以便了解对象所属 `bucket`、`key`、大小和状态等基础属性。

验收标准：

- 页面提供对象元数据列表入口
- 页面提供对象元数据详情入口
- 查询接口对齐 `GET /api/v1/objects`
- 详情接口对齐 `GET /api/v1/objects/{object_id}`
- 页面字段口径对齐 `StorageObject`

### US-003: 创建对象元数据

作为租户用户，我希望创建对象元数据，以便在对象资源目录中登记对象条目。

验收标准：

- 页面提供创建对象元数据入口
- 创建接口对齐 `POST /api/v1/objects`
- 创建请求体包含 `bucket`、`key`、`idempotency_key`
- 可选字段只承接 `size_bytes`、`content_type`
- 页面明确该动作是“元数据创建”，不是文件上传
- 成功返回口径对齐 `201 + StorageObject`

### US-004: 删除对象元数据

作为租户用户，我希望删除不再使用的对象元数据，以便清理租户内资源目录。

验收标准：

- 页面提供删除对象元数据入口
- 删除接口对齐 `DELETE /api/v1/objects/{object_id}`
- 页面在删除前提示关键引用风险
- 文档不把 `409 CONFLICT` 写成当前已冻结删除契约
- 删除返回口径对齐当前 `Core v1.yaml`

### US-005: 保持对象存储模块的 Core 边界清晰

作为模块维护者，我希望对象存储正文只声明当前 `Core` 已冻结的对象元数据资源和待补能力边界，以便后续与 `Core` 团队对齐时没有歧义。

验收标准：

- 页面正文明确对象资源归 `Core`
- 页面正文不继续使用旧 `/api/v1/storage/*` 或 `/api/v1/console/*` 路径
- 页面正文不要求前端显式传 `tenant_id / X-Tenant-Id`
- 页面正文明确上传、下载、桶策略、桶资源当前不是已冻结 Core 路径
- 页面正文不把对象资源写入 `Services /api/v1/svc/*`

## 5. 功能需求

- FR-1：系统必须提供对象存储页面总览，并明确已冻结能力与待补能力边界
- FR-2：系统必须提供对象元数据的查询、详情、创建和删除入口
- FR-3：系统必须从认证上下文获取租户边界，不依赖前端传入 `tenant_id`
- FR-4：系统必须对创建接口应用 `idempotency_key`
- FR-5：系统必须统一使用标准错误结构
- FR-6：系统必须明确当前页面只承接对象元数据，不承接实际上传 / 下载链路
- FR-7：系统必须明确删除动作当前仅承接 `Core v1.yaml` 已冻结返回码
- FR-8：系统必须让最终正文可直接作为对象存储模块对齐 `Core` 的主维护材料

## 6. 非目标

- 不在本轮把对象上传写成已冻结 Core 契约
- 不在本轮把对象下载写成已冻结 Core 契约
- 不在本轮把桶资源、桶列表、桶策略写成已冻结 Core 契约
- 不在本轮实现对象版本管理、生命周期规则、预签名 URL 等额外动作
- 不在本轮实现 `BOSS` 侧平台对象存储池运营总览
- 不在本轮改变现有 `Core v1.yaml` 已冻结的返回码和主路径

## 7. 产品设计考虑

- 页面口径必须是当前租户权限范围内的对象元数据，不能写成全平台对象存储池视角
- 总览页可保留上传、下载、桶策略等产品规划语义，但必须标记为待补能力
- `VM`、容器、`GPU 容器` 页面里的“对象”入口应回指到本模块已冻结的对象元数据定义
- 删除前应提示关键引用风险
- 页面需明确 `bucket` 当前只是对象字段，不代表已存在单独的桶资源管理页契约

## 8. 成功标准

- 用户能在 30 秒内识别当前租户下有哪些对象元数据资源
- 用户能在同一模块内完成对象元数据的基础创建和查询
- 文档中关于对象存储的路径、返回码、字段与现有 `Core v1.yaml` 不再冲突
- 文档不再把上传、下载、桶策略、桶资源误写成已经冻结的 Core 契约

## 9. 开放问题

- 对象上传 / 下载后续是否由独立对象网关路径承担
- 桶资源后续是否作为独立 `Core` 路径组补充
- 删除冲突语义是否需要后续在 `Core v1.yaml` 中正式冻结

## 10. 回填前置依赖

- 后续若要把对象上传 / 下载写成正式接口，需先补充到 `Core v1.yaml`
- 后续若要补桶资源或桶策略，应优先走 `Core v1.yaml` 扩充
- 若页面需要展示更细粒度的对象引用关系，应先确认 `StorageObject` 当前 schema 是否足够支撑前端展示
