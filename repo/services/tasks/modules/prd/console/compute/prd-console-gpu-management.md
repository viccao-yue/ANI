# PRD: Console GPU 算力管理

## 1. 页面定位

`Console / 算力与云资源 / GPU 算力管理` 是租户侧资源观察与排障页面，用于统一表达当前租户权限范围内的 GPU 资源视图、占用关系、异常定位入口和后续待补动作边界。

本页属于 `Console` 页面，底层资源归属仍然是 `Core`，不得把 GPU 资源对象写成 `Services` 业务契约，也不得混入 `BOSS` 的全平台运营口径。

## 2. 权威源结论

当前一级权威源以 `ANI-main/repo/api/openapi/v1.yaml` 为准，本模块需要先明确以下事实：

- `GPU` 在 `Core` 总体资源域说明中出现，说明其归属属于 `Core`
- `InstanceRecord` 中存在 `gpu` 字段，但该字段用于 `gpu_container` 实例摘要，不等于已经冻结独立的 GPU 资源页接口
- 当前权威源**没有**冻结独立的 `GPU overview / node / device / occupancy / allocation / release` API 路径
- 当前权威源**没有**声明 `GpuOverviewResponse`、`GpuDistributionResponse`、`GpuNodeListResponse`、`GpuDeviceListResponse`、`GpuOccupancyResponse` 等 schema

因此，本轮 PRD 的目标不是伪造 GPU 独立 API，而是把页面职责、展示边界、待补能力和与现有 `Core` 资源域的关系收口清楚。

## 3. Goals

- 让租户用户理解 `GPU 算力管理` 页应展示什么、为什么展示、与哪些模块联动
- 明确本页只承接 `Core` GPU 资源域，不写入 `Services`
- 明确哪些能力已可安全表述，哪些能力仍属于 `Core` 待补边界
- 为后续 `Core` 冻结 GPU 独立接口时提供稳定的产品定义基线

## 4. 用户故事

### US-001: 查看 GPU 总览入口

作为租户用户，我希望进入页面后能快速判断 GPU 是否紧张、是否存在异常，以便决定是否继续排查。

验收标准：

- 页面首屏包含 `GPU 总量`、`已分配`、`空闲`、`平均利用率`、`异常设备数` 这类目标区块定义
- 每个区块都要求标明统计窗口、刷新时间或“数据暂不可用”
- 页面口径限定为当前租户权限范围，不使用平台资源池排行表达
- 当前若权威源尚未冻结对应接口，页面文档必须明确标注为 `待 Core 补齐`

### US-002: 查看型号、节点和设备层级

作为租户用户，我希望从总览继续下钻到型号、节点和设备层级，以便定位异常资源。

验收标准：

- 页面结构中包含 `GPU 型号分布`、`GPU 节点列表`、`GPU 设备列表`
- 文案明确节点/设备是资源明细层，不是平台运营排行
- 详情和明细只覆盖当前租户有权限查看的对象
- 文档不得自造正式路径、正式 schema 或正式返回码

### US-003: 查看租户内占用关系

作为租户用户，我希望知道当前 GPU 被哪些项目、命名空间、工作负载或实例占用，以便排查资源紧张原因。

验收标准：

- 页面包含 `租户内占用分布` 区块
- 占用关系只表达当前租户内部视角，不写成跨租户排行
- 占用区块允许跳转到 `VM`、`容器实例`、`GPU 容器实例` 等关联模块
- 当前若权威源尚未冻结占用接口，只能写成页面目标和待补边界

### US-004: 理解动作边界

作为有权限的租户用户，我希望知道页面未来是否承接 GPU 分配、回收或释放动作，以及当前阶段能否直接执行。

验收标准：

- 页面可以保留 `资源动作` 区块或入口位
- 文档必须明确：当前权威源尚未冻结 GPU 独立写接口，不能把动作入口写成已可调用的正式 API
- 若后续 `Core` 冻结写操作，应满足 `idempotency_key`、统一错误结构和明确成功状态码
- 当前阶段如需动作联动，只能表达为“待补能力”或“跳转到关联资源操作”

### US-005: 保持 Core 边界清晰

作为模块维护者，我希望 GPU 模块正文只写已证实事实和待补边界，避免把未来设计误写成当前契约。

验收标准：

- 文档明确 `GPU` 资源归 `Core`
- 文档不继续使用旧 `/api/v1/console/*` 路径
- 文档不要求前端显式传 `tenant_id` 或 `X-Tenant-Id`
- 文档不自造接口路径、schema、错误码和返回码

## 5. 功能需求

- FR-1：系统必须把 `GPU 算力管理` 定义为租户侧资源观察与排障页面
- FR-2：系统必须在页面结构中保留总览、型号分布、节点列表、设备列表、占用分布等区块定义
- FR-3：系统必须把页面视角限定为当前租户权限范围
- FR-4：系统必须把 `GPU` 资源明确归类到 `Core`
- FR-5：系统必须明确当前权威源尚未冻结独立 GPU 资源 API，正文不得自造正式接口
- FR-6：系统必须为后续 `Core` 冻结独立 GPU 接口预留稳定的产品定义边界
- FR-7：系统必须支持与 `VM`、`容器实例`、`GPU 容器实例`、`平台概览` 的联动说明
- FR-8：系统必须统一使用标准错误结构口径：`{"code":"UPPER_SNAKE","message":"...","request_id":"..."}`

## 6. 非目标

- 不在本轮定义 GPU 独立查询或写操作的正式 API 路径
- 不在本轮编造 GPU 独立 schema、operationId 或返回码
- 不在本轮展开底层调度算法、设备插件和平台资源池运营策略
- 不在本轮把 GPU 资源契约写入 `Services /api/v1/svc/*`

## 7. 产品设计考虑

- 页面语言优先使用 `当前租户可见资源`、`占用分布`、`异常设备`，避免 `全平台资源池态势`
- 即使当前独立 API 未冻结，也要先把页面结构、字段口径和跳转关系定义稳定
- 动作入口如保留，必须显式标注为 `待补能力`，不能伪造成已实现能力
- 页面异常状态要区分 `资源异常`、`监控缺失`、`能力待补`

## 8. 成功标准

- 用户能从文档直接理解 GPU 页面与其他实例类模块的关系
- 模块文档不再包含任何伪造的 GPU 独立 API 路径和 schema
- 页面目标、待补边界和 Core 归属表达一致，不再混淆 `Core / Services`

## 9. 开放问题

- `Core` 后续是否会冻结独立 GPU inventory 路径，还是继续通过其他资源域聚合承接
- 节点级低层监控指标是否允许租户直接查看
- GPU 分配/释放是否会作为独立资源动作冻结，还是通过实例相关能力间接体现

## 10. 回填前置依赖

- 若后续要把 `GPU 总览 / 节点 / 设备 / 占用分布` 写成正式接口，必须先在 `ANI-main/repo/api/openapi/v1.yaml` 中冻结对应路径和 schema
- 若后续要把 GPU 分配或释放写成正式动作，必须先冻结成功返回码、错误码、`requestBody.required` 和幂等要求
- 在上述权威源补齐前，`PRD / SPEC / 主维护文档 / HTML` 只能把这些能力写成待补边界，不能写成正式契约
