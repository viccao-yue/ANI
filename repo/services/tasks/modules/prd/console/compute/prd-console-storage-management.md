# PRD: Console 存储管理

## 1. 页面定位

`Console / 算力与云资源 / 存储管理` 用于帮助租户用户查看、创建、删除和理解当前权限范围内的存储资源，并为 `VM`、`容器`、`GPU 容器` 等工作负载提供统一的存储关联入口。

本页属于 `Console` 租户侧资源管理页面，不是 `BOSS` 的平台存储池运营页。

## 2. 权威源结论

当前一级权威源以 `ANI-main/repo/api/openapi/v1.yaml` 为准，可确认：

- `StorageVolume`、`StorageFilesystem`、`StorageObject`、`VectorStore` 资源归 `Core`
- 已冻结的正式路径包括：
  - `/api/v1/volumes`
  - `/api/v1/filesystems`
  - `/api/v1/objects`
  - `/api/v1/vector-stores`
- 已冻结的向量存储动作包括：
  - `POST /api/v1/vector-stores/{vector_store_id}/search`
- 当前**尚未冻结**：
  - 卷快照
  - 卷挂载 / 卸载
  - 对象上传 / 下载
  - 桶资源 / 桶策略
  - 文件系统挂载目标

## 3. Goals

- 让租户在一个页面内理解并管理当前权限范围内的存储资源
- 严格以现有 `Core v1.yaml` 为准，沉淀存储管理页面可直接用于 `Core` 对齐的主维护材料
- 清楚区分哪些存储能力已经冻结，哪些仍为待补能力
- 为 `VM`、容器类页面提供统一的存储关联回指入口
- 保持 `Console` 租户视角，不混入平台存储池运营语义

## 4. 用户故事

### US-001: 查看存储管理总览

作为租户用户，我希望在一个存储管理页面里快速理解块存储、文件存储、对象存储和向量存储之间的边界，以便找到正确的存储资源。

验收标准：

- 页面明确区分 `块存储 / 文件存储 / 对象存储 / 向量存储`
- 页面明确标出哪些能力已经在 `Core v1.yaml` 冻结
- 页面明确标出 `快照 / 挂载 / 上传下载 / 桶策略 / 挂载目标` 仍为待补能力
- 页面不混入平台侧全局存储池口径

### US-002: 查看和创建块存储卷

作为租户用户，我希望查看和创建块存储卷，以便为实例准备基础持久化卷资源。

验收标准：

- 页面提供块存储卷列表和详情入口
- 页面提供创建块存储卷入口
- 查询接口对齐 `GET /api/v1/volumes`
- 创建接口对齐 `POST /api/v1/volumes`
- 创建请求体包含 `idempotency_key`
- 删除接口对齐 `DELETE /api/v1/volumes/{volume_id}`
- 页面不把挂载 / 卸载和快照动作误写成已冻结接口

### US-003: 查看和创建文件存储

作为租户用户，我希望查看和创建文件存储，以便为多个工作负载提供共享文件能力。

验收标准：

- 页面提供文件存储列表和详情入口
- 页面提供创建文件存储入口
- 查询接口对齐 `GET /api/v1/filesystems`
- 创建接口对齐 `POST /api/v1/filesystems`
- 创建请求体包含 `idempotency_key`
- 删除接口对齐 `DELETE /api/v1/filesystems/{filesystem_id}`
- 页面只承接当前已冻结字段，不扩写挂载目标管理子路径

### US-004: 查看和创建对象元数据

作为租户用户，我希望查看和创建对象元数据记录，以便在平台内管理对象资源清单和引用关系。

验收标准：

- 页面提供对象元数据列表和详情入口
- 页面提供创建对象元数据入口
- 查询接口对齐 `GET /api/v1/objects`
- 创建接口对齐 `POST /api/v1/objects`
- 创建请求体包含 `idempotency_key`
- 删除接口对齐 `DELETE /api/v1/objects/{object_id}`
- 页面明确当前只对齐“对象元数据”资源，不把上传 / 下载动作写成已冻结接口

### US-005: 查看、创建和搜索向量存储

作为租户用户，我希望查看、创建和搜索向量存储，以便为知识库和语义检索场景准备底层向量资源。

验收标准：

- 页面提供向量存储列表和详情入口
- 页面提供创建向量存储入口
- 页面提供向量搜索入口
- 查询接口对齐 `GET /api/v1/vector-stores`
- 创建接口对齐 `POST /api/v1/vector-stores`
- 搜索接口对齐 `POST /api/v1/vector-stores/{vector_store_id}/search`
- 创建请求体包含 `idempotency_key`

### US-006: 保持存储模块的 Core 边界清晰

作为模块维护者，我希望存储管理正文只声明当前 `Core` 已冻结的存储资源和待补能力边界，以便后续与 `Core` 团队对齐时没有歧义。

验收标准：

- 页面正文明确存储资源归 `Core`
- 页面正文不继续使用旧 `/api/v1/storage/*` 或 `/api/v1/console/*` 路径
- 页面正文不要求前端显式传 `tenant_id / X-Tenant-Id`
- 页面正文明确 `快照 / 挂载 / 上传下载 / 桶策略 / 挂载目标` 当前不是已冻结 Core 路径
- 页面正文不把存储资源写入 `Services /api/v1/svc/*`

## 5. 功能需求

- FR-1：系统必须提供存储管理总览，并明确各资源边界
- FR-2：系统必须提供块存储卷的查询、创建、详情和删除入口
- FR-3：系统必须提供文件存储的查询、创建、详情和删除入口
- FR-4：系统必须提供对象元数据的查询、创建、详情和删除入口
- FR-5：系统必须提供向量存储的查询、创建、详情、删除和搜索入口
- FR-6：系统必须从认证上下文获取租户边界，不依赖前端传入 `tenant_id`
- FR-7：系统必须对创建类接口应用 `idempotency_key`
- FR-8：系统必须统一使用标准错误结构
- FR-9：系统必须明确哪些存储能力已冻结、哪些仍待补
- FR-10：系统必须让最终正文可直接作为存储模块对齐 `Core` 的主维护材料

## 6. 非目标

- 不在本轮把卷快照写成已冻结 Core 契约
- 不在本轮把卷挂载 / 卸载动作写成已冻结存储接口
- 不在本轮把对象上传、下载、桶策略写成已冻结 Core 契约
- 不在本轮把文件系统挂载目标管理写成已冻结 Core 契约
- 不在本轮实现 `BOSS` 侧平台存储池运营总览
- 不在本轮改变现有 `Core v1.yaml` 已冻结的返回码和主路径

## 7. 产品设计考虑

- 页面口径必须是当前租户权限范围内的存储资源，不能写成全平台存储池视角
- 总览页可保留快照、挂载、对象上传下载等产品规划语义，但必须标记为待补能力
- `VM`、容器、`GPU 容器` 页面里的 `卷 / 文件 / 对象 / 向量库` 入口应回指到本模块已冻结的存储资源定义
- 风险操作应提示引用关系和删除影响

## 8. 成功标准

- 用户能在 30 秒内识别当前租户下有哪些存储资源
- 用户能在同一模块内完成已冻结存储资源的基础创建和查询
- 文档中关于存储模块的路径、返回码、字段与现有 `Core v1.yaml` 不再冲突
- 文档不再把待补能力误写成已经冻结的 Core 契约

## 9. 开放问题

- 卷快照后续是否作为独立 `Core` 路径组补充，还是继续附着在实例生命周期动作之下
- 对象上传 / 下载是否会进入独立 `Core` 动作路径，还是只保留对象元数据资源
- 文件系统挂载目标是否后续需要补入 `Core v1.yaml`

## 10. 回填前置依赖

- 后续若要把快照写成正式接口，需先补充到 `Core v1.yaml`
- 后续若要补挂载 / 卸载、对象上传 / 下载、桶策略、文件系统挂载目标，应优先走 `Core v1.yaml` 扩充
- 若页面需要展示更细粒度的引用关系，应先确认 `StorageVolume / StorageFilesystem / StorageObject / VectorStore` 当前 schema 是否足够支撑前端展示
