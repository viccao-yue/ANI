# PRD: Console 平台概览

## 1. Introduction/Overview

`Console / 首页与总览 / 平台概览` 是租户进入 Console 后的默认聚合首页，用于在一个页面内汇总当前工作区最重要的资源、服务、知识库、风险和待处理事项，帮助用户快速判断当前环境是否健康，并进入对应模块继续处理。

本模块是 `Console` 侧聚合页，不单独改变底层资源归属：

- `Core` 资源类事实仍来自各自已冻结模块
- `Services` 业务类事实仍来自各自已冻结模块
- 当前阶段不把首页聚合能力写成已冻结独立接口

## 2. Goals

- 让用户在一次进入首页后识别当前工作区最重要的风险项
- 在同页展示 `资源使用概览`、`GPU 利用率`、`推理服务状态`、`知识库调用趋势`、`告警与待处理事项`
- 为每个主题区块提供明确的明细跳转入口
- 严格保持 `Core / Services` 分层边界，不把首页聚合写成新的冻结资源契约
- 明确首页的空态、局部失败态、无权限态和刷新时间说明

## 3. User Stories

### US-001: 快速识别当前工作区健康状态
**Description:** 作为租户管理员或工作区 Owner，我希望进入首页后立即知道当前环境是否存在阻断问题，以便优先处理风险。

**Acceptance Criteria:**
- [ ] 首页首屏展示当前工作区状态总判断
- [ ] 用户无需进入多个子模块即可识别最重要风险
- [ ] 阻断问题与一般告警在视觉优先级上有明确区分

### US-002: 从摘要区块进入明细模块
**Description:** 作为日常使用者，我希望从首页摘要直接跳转到对应模块继续处理，而不是重新寻找入口。

**Acceptance Criteria:**
- [ ] 每个主题区块都提供明细入口
- [ ] 跳转目标限定为当前可用的 Console 模块
- [ ] 跳转后不丢失基本上下文

### US-003: 在局部异常场景下继续使用首页
**Description:** 作为租户用户，我希望即使某个区块的数据失败，首页其余区块仍可继续使用。

**Acceptance Criteria:**
- [ ] 单区块失败时仅该区块展示错误提示和刷新入口
- [ ] 其余区块继续渲染，不整页阻断
- [ ] 页面明确显示刷新时间或统计时间窗口

### US-004: 严格遵循 Core 标准
**Description:** 作为模块维护人，我希望平台概览文档完全遵循 `Core` 规范，不编造首页专属契约。

**Acceptance Criteria:**
- [ ] 文档不再使用旧 `/api/v1/console/home/*` 路径
- [ ] 文档不把 `/api/v1/svc/console/platform-overview` 写成当前已冻结事实
- [ ] 文档不自造首页专属 schema 名称
- [ ] 文档明确租户边界来自认证上下文，而不是前端显式传入 `tenant_id`

## 4. Functional Requirements

- FR-1: 系统必须在首页展示工作区总体状态判断
- FR-2: 系统必须展示 5 大主题区块的摘要信息
- FR-3: 系统必须允许区块级跳转到当前可用模块
- FR-4: 系统必须支持空态、局部失败态和无权限态
- FR-5: 系统必须显示刷新时间或统计时间窗口
- FR-6: 系统必须严格区分 `Core` 数据与 `Services` 数据来源
- FR-7: 系统不得把首页聚合逻辑写成新的正式资源契约

## 5. Non-Goals (Out of Scope)

- 不在当前阶段冻结独立首页聚合接口
- 不在当前阶段定义首页专属返回 schema
- 不在首页直接承接复杂创建、删除、编辑类操作
- 不在首页替代各子模块的完整状态机、筛选器和批量操作能力
- 不把未完成模块域强行写成正式首页入口

## 6. Design Considerations

- 首页文案必须强调“摘要与导航”，不冒充“全量明细页”
- 区块状态命名必须和来源模块保持一致
- 无数据与真实值为 0 必须区分
- 首页高风险提示不等于审计页、告警中心或运维控制台

## 7. Technical Considerations

- 当前首页允许前端并发调用已冻结子模块接口，并在页面层做轻聚合
- 如果未来需要后端聚合能力，必须先在权威源中冻结路径、返回码和 schema
- 首页不应要求前端显式传 `tenant_id / X-Tenant-Id`
- 错误结构统一遵循 `{"code":"UPPER_SNAKE","message":"...","request_id":"..."}`

## 8. Success Metrics

- 用户能在 30 秒内判断当前工作区是否存在阻断风险
- 用户能在首页直接进入最需要处理的模块
- 首页在单区块异常时仍保持可用
- 文档口径与 `Core / Services` 边界保持一致，不出现自造契约

## 9. Open Questions

- 首页是否需要未来新增后端只读聚合接口
- 告警与待处理事项的摘要来源是否需要单独模块化
- 区块跳转是否需要统一携带预置筛选条件

## 10. 回填前置依赖

- 如未来新增首页后端聚合能力，必须先在权威源中冻结只读路径和 schema
- 告警与待处理事项若要独立成模块，需先明确其实际冻结资源域
- 首页跳转若要携带统一筛选参数，需先与对应子模块确认参数口径
