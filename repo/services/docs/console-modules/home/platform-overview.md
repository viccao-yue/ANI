# 平台概览

## 页面定位

平台概览是 `Console` 首页的聚合页，用于把租户当前最重要的信息集中展示在一个页面内，帮助用户快速判断：

- 当前租户环境是否正常
- 资源是否充足
- 推理服务是否健康
- 知识库是否有活跃访问或异常
- 是否存在需要立刻处理的异常或待办事项

以上信息统一合并到一个 `平台概览` 页面内，不拆成 5 个独立首页页面。

## 文档管理规则

- 本文是 `平台概览` 的主维护文档，首页定位、聚合边界、字段口径、跳转规则和验收标准统一以本文为准
- 如 `PRD`、`SPEC`、HTML 摘要与本文出现差异，先对照 `tasks/modules/prd/console/home/prd-console-platform-overview.md`、`tasks/modules/spec/console/home/spec-console-platform-overview.md` 和当前可用模块的冻结事实，再统一回写
- 平台概览当前没有独立冻结接口、独立 schema 和独立返回码，因此正文不得自造首页专属契约
- 首页只能指向当前已判定为 `可用` 的 Console 模块入口，不得把尚未落地的“监控页 / 记录页 / 分析页”写成既有跳转目标

## Core 层要求

- 本页属于 `Services` 层的 Console UI 聚合能力
- 页面内展示的数据可同时来自 `Core` 和 `Services`
- `Core` 资源类接口必须使用 `/api/v1/*`
- `Services` 业务类接口必须使用 `/api/v1/svc/*`
- 不允许继续使用旧的 `/api/v1/console/home/*` 路径
- 页面可以做聚合展示，但不能把 Core 资源错误写进 Services 契约里
- 首页不要求前端显式传 `tenant_id / X-Tenant-Id`
- 首页错误结构统一遵循 `{"code":"UPPER_SNAKE","message":"...","request_id":"..."}`

## 页面职责

- 负责概览与跳转，不替代各明细模块
- 负责摘要与趋势，不直接承载复杂管理操作
- 负责异常提醒与待处理事项摘要，不替代对应模块内的明细页和处置流程

## 5 大主题区块

| 区块 | 展示目标 | 主要内容 |
|---|---|---|
| 资源使用概览 | 快速判断资源是否紧张 | 实例数、GPU 使用、存储占用、网络/集群摘要 |
| GPU 利用率 | 快速识别算力是否闲置或过载 | GPU 总量、已分配、平均利用率、异常资源 |
| 推理服务状态 | 快速发现部署失败或异常服务 | 运行中、部署中、失败、最近活跃情况 |
| 知识库调用趋势 | 判断知识库使用热度和波动 | 查询量趋势、最近调用、异常解析情况 |
| 告警与待处理事项 | 把最需要处理的问题推到首页 | 高优先级告警、处理中任务、失败任务、待确认事项 |

```text
平台概览
├── 顶部总判断
│   └── 当前工作区状态总览
├── 5 大主题区块
│   ├── 资源使用概览
│   ├── GPU 利用率
│   ├── 推理服务状态
│   ├── 知识库调用趋势
│   └── 告警与待处理事项
└── 跳转入口
    ├── 算力与云资源
    ├── 推理服务
    ├── 知识库与 AI 应用
    ├── 用量与计量
    └── 安全与身份（如有权限）
```

## 交互与使用规则

- 首页默认进入平台概览，不要求用户先切换多个模块再判断当前工作区状态
- 5 个主题区块统一在本页展示，不再拆成 5 个独立首页页面
- 每个区块都应提供“查看详情”或等价跳转入口
- 平台概览只展示摘要信息，不在首页直接放复杂编辑类操作
- 当某个模块数据缺失时，允许局部空态或局部错误态，但不应让整页不可用
- 页面必须展示数据刷新时间或聚合时间窗口，避免用户误把聚合值当实时值

## 数据来源与分层约束

### 数据来源划分

- Core 数据（必须走 `/api/v1/*`）
  - 实例 / VM
  - GPU 资源
  - 网络 / 存储 / K8s
  - 计量 / 用量
- Services 数据（必须走 `/api/v1/svc/*`）
  - 推理服务
  - 知识库
  - 模型相关摘要（若首页需要）

### 首页接口策略

- 当前不冻结独立的 `平台概览` 聚合接口
- 当前可行方式：前端并发读取各已冻结子模块接口，并在页面层做轻聚合
- 若未来需要降低首页并发调用成本，可补充只读聚合能力，但该能力在当前阶段仍属于待补项
- 不推荐继续使用：`/api/v1/console/home/overview`

### 关键边界

- 首页可以展示 Core 数据，但不意味着这些资源属于 Services
- 首页可以展示 Services 数据，但不应把其实现逻辑错误下沉到 Core
- 页面中出现的路径、状态、错误码命名必须和对应模块保持一致
- 若某项数据当前口径未冻结，应明确标注“待确认”，不得伪造最终值

## 5 大主题区块详细说明

### 资源使用概览

- 展示重点：实例总数、运行中实例数、GPU 已分配量、存储已使用量、网络/集群资源摘要
- 主要来源层：Core
- 展示口径：展示当前租户下资源摘要，强调总量、已用量、余量或使用率，不替代资源明细页
- 异常/空态：若某项资源暂未接入，则区块局部展示“待接入”或“暂无数据”，不能让整块消失
- 跳转目标：算力与云资源 / 网络管理 / 存储管理 / 集群相关页面
- **明细详文**：`home-resource-overview.md`

### GPU 利用率

- 展示重点：GPU 总量、已分配量、平均利用率、空闲资源、异常 GPU 数
- 主要来源层：Core
- 展示口径：优先展示窗口聚合值，不要求逐秒实时；必须说明统计周期和时间窗口
- 异常/空态：当 GPU 监控未回传时，应展示“监控暂不可用”并保留跳转入口
- 跳转目标：GPU 算力管理
- **明细详文**：`home-gpu-utilization.md`

### 推理服务状态

- 展示重点：运行中、部署中、失败、已停止、最近活跃服务数
- 主要来源层：Services
- 展示口径：状态命名必须与推理服务模块保持一致，首页只做汇总，不改写状态机
- 异常/空态：若服务列表为空，可展示“当前无推理服务”；若最近有失败实例，应高亮异常数量
- 跳转目标：推理服务
- **明细详文**：`home-inference-status.md`

### 知识库调用趋势

- 展示重点：调用量趋势、成功率、最近活跃知识库数、异常解析或失败调用数
- 主要来源层：Services
- 展示口径：必须明确时间范围和粒度，例如近 24 小时 / 近 7 天 / 按小时聚合
- 异常/空态：若租户未创建知识库，可展示“暂无知识库数据”；若调用异常升高，应在首页提示
- 跳转目标：知识库管理 / 问答（`home-kb-trend.md` / `kb-qa-flow.md`）

### 告警与待处理事项

- 展示重点：高优先级告警数、处理中任务数、失败任务数、待确认事项数
- 主要来源层：Core + Services
- 展示口径：首页只展示最需处理的摘要，不承担完整筛选、批量处理和审计处置能力
- 异常/空态：无告警时可展示“当前无高优先级告警”；有阻断性问题时应在首页首屏高亮
- 跳转目标：失败来源模块 / 任务中心 / 安全与身份概览 / 对应模块
- **明细详文**：`../alerts/alerts-pending-items.md`；任务见 `../alerts/async-task-center.md`

## 字段级定义

### 资源使用概览

| 字段 | 说明 | 展示建议 |
|---|---|---|
| 实例总数 | 当前租户下实例资源总量 | 整数；显示总数 |
| 运行中实例数 | 当前处于 running 状态的实例数量 | 整数；可附占比 |
| GPU 已分配量 | 已被实例、任务或服务占用的 GPU 资源 | 显示已分配 / 总量 |
| 存储已使用量 | 当前租户已占用的存储容量 | 显示数值 + 单位，如 TB / GB |
| 资源更新时间 | 本区块数据最新刷新时间 | 显示绝对时间或“xx 分钟前” |

### GPU 利用率

| 字段 | 说明 | 展示建议 |
|---|---|---|
| GPU 总量 | 可供当前租户使用或可见的 GPU 总资源量 | 整数 |
| 已分配 GPU | 已被分配出去的 GPU 数量 | 显示已分配 / 总量 |
| 平均利用率 | 统计窗口内 GPU 平均使用率 | 百分比；需标明时间窗口 |
| 空闲 GPU | 当前可继续使用的空闲 GPU 数量 | 整数；可做绿色提示 |
| 异常 GPU 数 | 监控异常、不可调度或状态异常的 GPU 数量 | 整数；大于 0 时高亮 |

### 推理服务状态

| 字段 | 说明 | 展示建议 |
|---|---|---|
| 运行中服务数 | 当前状态为运行中的推理服务数量 | 整数 |
| 部署中服务数 | 处于发布、启动、灰度或初始化阶段的服务数量 | 整数；可做进行中提示 |
| 失败服务数 | 部署失败或健康检查失败的服务数量 | 整数；大于 0 时高亮 |
| 已停止服务数 | 当前已停用或下线的服务数量 | 整数 |
| 最近活跃服务数 | 最近一个统计窗口内有调用记录的服务数量 | 整数；需说明统计窗口 |

### 知识库调用趋势

| 字段 | 说明 | 展示建议 |
|---|---|---|
| 调用量趋势 | 统计周期内知识库请求量变化 | 折线图或简化趋势图 |
| 调用成功率 | 成功请求占总请求的比例 | 百分比；低于阈值时提示异常 |
| 最近活跃知识库数 | 最近一段时间有调用的知识库数量 | 整数 |
| 异常解析数 | 解析失败、召回异常或处理异常的次数 | 整数；大于 0 时高亮 |
| 统计时间范围 | 当前趋势图对应的时间窗口 | 例如近 24 小时 / 近 7 天 |

### 告警与待处理事项

| 字段 | 说明 | 展示建议 |
|---|---|---|
| 高优先级告警数 | 当前待处理的高优先级告警数量 | 整数；首屏高亮 |
| 处理中任务数 | 当前尚未完成的处理中任务数量 | 整数 |
| 失败任务数 | 执行失败且需要人工关注的任务数量 | 整数；大于 0 时高亮 |
| 待确认事项数 | 等待人工确认、审批或补充信息的事项数量 | 整数 |
| 最近更新时间 | 本区块聚合结果生成时间 | 显示刷新时间，避免误解为实时流式数据 |

## 字段展示规则

| 场景 | 展示规则 | 说明 |
|---|---|---|
| 正常 | 显示指标值 + 更新时间 + 查看详情入口 | 首页默认状态 |
| 暂无数据 | 显示暂无数据文案，不显示伪造的 0 值 | 适用于未接入或当前无数据 |
| 待确认口径 | 显示待确认标签，不固化错误字段定义 | 适用于口径尚未冻结的指标 |
| 局部异常 | 仅高亮异常字段，不让整页不可用 | 例如某一个来源接口超时 |
| 阻断异常 | 在首页显著提示，并保留跳转到故障来源页 | 例如高优先级告警或关键服务失败 |

## 字段口径与单位

| 字段 | 口径建议 | 单位/格式 |
|---|---|---|
| 实例总数 / 运行中实例数 | 以当前租户可见实例为准，不跨租户汇总 | 整数 |
| GPU 已分配量 / GPU 总量 / 空闲 GPU | 以当前租户可见资源池或已授权资源范围为准 | 整数或卡数 |
| 平均利用率 | 按约定时间窗口聚合，不写成瞬时值 | 百分比，保留 0~2 位小数 |
| 存储已使用量 | 以租户当前已占用容量为准，不与物理总盘混淆 | GB / TB |

## 产品经理补充定义

### 目标用户与权限视角

- 租户管理员与工作区 Owner：查看全页摘要、进入各明细模块、处理需要人工确认的事项
- 业务成员：查看与自身权限范围一致的摘要卡片，并进入对应明细页完成后续动作
- 只读用户：查看概览与风险提示，但不在首页承接任何高风险写操作

### 默认视图与页面状态

- 首屏默认展示 `顶部总判断 + 5 大主题区块 + 快速跳转入口`，不要求用户先进入各子模块才能判断当前工作区状态
- 当租户尚未创建任何资源时，页面展示“空工作区”引导，并保留去 `VM / GPU / 推理服务 / 知识库 / 安全与身份` 的入口
- 当单个来源接口失败时，仅该区块显示局部错误和刷新入口，其他区块继续可用
- 首页必须显示聚合时间窗口或刷新时间，避免把聚合结果误解为实时值

### 核心任务流

1. 用户进入首页后，先根据顶部总判断识别是否存在阻断问题
2. 用户点击异常区块或“查看详情”入口，带着当前筛选上下文进入对应子模块继续处理
3. 用户处理完成后返回首页，再次确认风险摘要是否恢复正常

### 跨模块协同

- 资源使用概览回跳到 `GPU 算力管理`、`VM`、`网络管理`、`存储管理`
- 服务状态与知识库趋势回跳到 `推理服务`、`知识库管理`、`租户用量报表`
- 安全或待办事项回跳到 `安全与身份概览`、`API Key 管理` 等对应模块

### 产品验收补充

- 用户必须能在一次进入首页后识别出当前工作区最重要的风险项
- 每个主题区块都必须提供可工作的明细入口，且跳转后不丢失基本上下文
- 空态、局部失败态、刷新时间说明必须同时具备，不能只保留正常态
- 首页只能承担摘要与导航，不得把子模块的复杂编辑流直接塞回首页

## 状态阈值建议

| 字段/场景 | 默认提示规则 | 说明 |
|---|---|---|
| 失败服务数 | 大于 0 即红色高亮 | 属于直接影响可用性的首页异常项 |
| 高优先级告警数 | 大于 0 即红色高亮 | 需要进入首屏可见区域 |
| 异常 GPU 数 | 大于 0 即黄色或红色提示 | 是否红色取决于是否影响可调度能力 |
| 调用成功率 | 低于约定阈值时黄色或红色提示 | 具体阈值待与对应模块口径确认 |
| 暂无数据 | 不显示正常绿态，不用 0 假装正常 | 必须与“真实值为 0”区分 |
| 待确认口径 | 显示待确认标签，不参与健康结论 | 避免误导 Core 和前端联调 |

## 字段来源归属

| 字段 | 来源层 | 说明 |
|---|---|---|
| 实例总数 / 运行中实例数 | Core | 属于实例资源摘要 |
| GPU 总量 / 已分配 GPU / 平均利用率 / 空闲 GPU / 异常 GPU 数 | Core | 属于 GPU 与资源监控摘要 |
| 存储已使用量 | Core | 属于存储资源摘要 |
| 运行中服务数 / 部署中服务数 / 失败服务数 / 最近活跃服务数 | Services | 属于推理服务业务摘要 |

| 高优先级告警数 / 处理中任务数 / 失败任务数 / 待确认事项数 | Core + Services | 允许多来源聚合，但首页只展示统一摘要 |

## 当前冻结事实

### 已冻结事实表

| 项 | 当前结论 | 说明 |
|---|---|---|
| 独立首页聚合接口 | 未冻结 | 当前不能把首页写成已有独立 `platform-overview` 接口 |
| 独立首页聚合 schema | 未冻结 | 当前不能自造 `ConsolePlatformOverviewResponse` 等 schema 名称 |
| 首页数据事实来源 | 已冻结子模块接口 | 首页摘要必须回指到各子模块已冻结路径 |
| 首页错误结构 | 统一错误结构 | 复用各来源模块统一错误结构 `{"code":"UPPER_SNAKE","message":"...","request_id":"..."}` |
| 首页跳转目标 | 当前可用模块入口 | 只能跳到当前已判定为 `可用` 的 Console 模块入口 |

### 当前允许的实现方式

- 前端并发调用已冻结的 `Core / Services` 子模块接口，在页面层做轻聚合
- 单个区块只读取与自身主题直接相关的已冻结资源，不跨域改写状态机
- 首页只消费摘要值、风险提示和跳转目标，不单独创造新的资源对象

## 创建前置条件

| 项 | 说明 |
|---|---|
| 本页定位 | 聚合页，**不承接**资源直接创建 |
| 创建入口 | 跳转至各子模块（VM / 容器 / 存储等）各自的创建流程 |

## 操作可用性矩阵

| 操作 | 只读用户 | 租户管理员 / Owner | 说明 |
|---|---|---|---|
| 查看首页摘要 | 可用 | 可用 | 首页默认能力 |
| 查看风险提示 | 可用 | 可用 | 不承接危险写操作 |
| 跳转到当前可用模块 | 可用 | 可用 | 仅跳转到当前状态板判定为 `可用` 的模块 |
| 首页直接创建 / 删除资源 | 不可用 | 不可用 | 当前明确不在首页承接 |
| 刷新区块摘要 | 可用 | 可用 | 允许局部刷新 |

## 接口冻结规则

### 独立首页接口状态

- 当前没有独立冻结的首页聚合路径
- 当前没有独立冻结的首页 schema
- 当前没有独立冻结的首页 operationId
- 当前没有独立冻结的首页专属返回码

### 来源模块依赖规则

- 首页只允许依赖当前可用模块的已冻结事实
- 首页不重复定义各子模块接口，而只引用其摘要结果
- 若某区块暂时没有可稳定回跳的子模块，只能回跳到当前失败来源模块或保留占位说明，不能自造新页面

## 错误示例

### 区块局部失败提示

```json
{
  "code": "UPSTREAM_UNAVAILABLE",
  "message": "gpu summary temporarily unavailable",
  "request_id": "req-home-503-001"
}
```

说明：

- 该错误结构只用于说明页面层必须沿用统一错误口径
- 当前不能把它写成首页独立冻结错误码契约

### 当前不允许写成既有事实的能力

- 不允许把 `/api/v1/svc/console/platform-overview` 写成当前已冻结路径
- 不允许把首页聚合返回体写成当前已冻结 schema
- 不允许把首页局部视图字段写成权威源中已存在的独立资源字段
- 不允许把首页错误码、状态枚举、top items 结构写成已经冻结的专属契约

## 待确认清单

| 待确认项 | 当前建议 | 需确认方 |
|---|---|---|
| overall_status 枚举值 | `healthy / warning / critical / unknown` | Core + Services |
| 资源区块状态阈值 | 先按失败、高优先级、异常资源数量做提示 | Core |
| avg_utilization 统计窗口 | 建议按 `hour` 聚合 | Core |
| success_rate 告警阈值 | 低于约定阈值时高亮 | Services |
| top_items 最多返回条数 | 建议首页只返回 3~5 条重点事项 | Core + Services |
| detail_url 最终跳转路径 | 只允许填写当前可用模块入口，否则置空 | 前端 + 产品 |
| summary.detail_url 是否保留 | 若没有独立总览落地页，建议直接置空 | 前端 + 产品 |

## 租户与权限边界

- 首页所有数据都必须基于当前登录租户返回，不能让前端通过 `tenant_id` 自行指定别的租户数据
- 如果后端提供聚合接口，租户信息应从认证上下文或中间件获取，不应信任 query 或 body 传入的 `tenant_id`
- 只有当前用户有权限看到的模块才允许在首页展示，无权限区块应返回隐藏、置灰或无权限提示
- 跨租户、跨项目、跨空间的数据口径必须在页面文案里明确，不允许把平台级指标误写成租户级指标
- 涉及管理员能力的入口只可做展示或跳转，不应默认暴露给普通租户角色

## 聚合实现边界

```text
浏览器 Console
    -> 当前已冻结子模块接口
        -> Core 资源摘要（VM / GPU / 网络 / 存储 / 计量）
        -> Services 业务摘要（推理服务 / 知识库 / 模型相关摘要）
    -> 页面层聚合为 5 大主题区块
```

约束说明：

1. 当前首页以已冻结子模块接口为事实来源，不单独声明新的聚合契约
2. 若后续确需补充后端聚合能力，应先在权威源中冻结只读聚合路径、返回码和 schema
3. 无论是否新增聚合层，都不能改变各子模块原始状态定义和资源归属

## 回填验收标准

| 验收项 | 通过标准 | 备注 |
|---|---|---|
| 路径合规 | 页面内不再出现旧的 `/api/v1/console/home/*` 路径 | 必须清晰区分 `/api/v1/*` 与 `/api/v1/svc/*` |
| 分层合规 | VM/GPU/网络/存储/计量明确标记为 Core 来源 | 不得误写到 Services 资源契约 |
| 聚合定位 | 页面文案明确这是 Services 层 Console 聚合页 | 但允许聚合 Core 与 Services 双侧数据 |
| 租户边界 | 文案明确租户信息来自认证上下文，不信任前端自传 `tenant_id` | 属于 Core 团队重点校验项 |
| 错误口径 | 错误响应格式写明 `code/message/request_id` | 需与 Core 文档保持一致 |
| 交付可追溯 | 每个区块都能说清来源层、跳转目标、展示边界 | 便于后续回填 YAML |

## 待确认项处理规则

- `overall_status` 等页面层字段在当前阶段只能作为产品层展示语义，不得写成已冻结 API 字段
- 若未来首页需要统一下钻参数，只能在对应模块确认后再升级为正式口径
- 若未来需要首页独立聚合 API，必须先冻结路径、schema、返回码和错误码，再回写本文

## 非目标与边界

- 本页不承担资源创建、删除、变更等复杂操作
- 本页不替代各明细模块的完整数据和完整状态机
- 本页不在当前阶段直接定义 Core 资源的新契约
- 本页不把 VM / GPU / 网络 / 存储错误写入 Services `v1.yaml`
- 本页当前只保留页面定义与 Core 合规要求，不继续承载 Phase 3 任务产物
