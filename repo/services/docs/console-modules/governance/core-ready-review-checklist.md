# Core-ready 严格复审清单

本文档用于在 `Console` 模块交付前执行最后一轮严格复审，目标不是“看起来差不多”，而是确保材料已经达到**可直接对齐 Core 的主维护标准**。

## 1. 使用方式

- 在模块 `PRD / SPEC / 主维护源 / HTML 摘要` 都完成后执行
- 复审顺序必须固定：
  1. 权威源对照
  2. 跨材料一致性检查
  3. 反自造契约检查
  4. 页面能力降噪检查
  5. 最终答复范围检查
- 只要命中任一“强制阻塞项”，就禁止输出“已达到 Core 层要求”

## 2. 权威源清单

复审时必须同时打开以下文件：

- `ANI-main/repo/api/openapi/v1.yaml`
- `ANI-main/repo/api/core-v1-compatibility-baseline.yaml`
- 当前模块 `PRD`
- 当前模块 `SPEC`
- 当前模块主维护源
- 当前模块 HTML 摘要

## 3. 第一轮：权威源对照

### 3.1 路径

- [ ] 当前模块所有“已冻结接口”都能在 `openapi/v1.yaml` 中找到
- [ ] 所有路径前缀正确区分 `Core /api/v1/*` 与 `Services /api/v1/svc/*`
- [ ] 不存在旧 `/api/v1/console/*`、旧 `/api/v1/storage/*`、旧历史路径残留

### 3.2 Schema

- [ ] 当前模块所有“已冻结 schema”都能在权威源中找到
- [ ] 未冻结能力没有被写成伪 schema、伪字段、伪状态
- [ ] 可选字段和开发态字段的处理规则已写清楚
- [ ] 若存在对象存储字段 `bucket`，已明确其是否只是对象字段，而不是独立桶资源

### 3.3 返回码与错误码

- [ ] 成功返回码与权威源一致
- [ ] 错误返回码与权威源一致
- [ ] 没有把未冻结返回码写成既有契约
- [ ] 错误结构统一为 `{"code":"UPPER_SNAKE","message":"...","request_id":"..."}`

### 3.4 鉴权与租户边界

- [ ] `security` 与权威源一致
- [ ] 没有要求前端显式传 `tenant_id / X-Tenant-Id`
- [ ] 租户边界明确由认证上下文/后端中间件获取

## 4. 第二轮：跨材料一致性检查

### 4.1 PRD

- [ ] `PRD` 没有保留与当前正文冲突的开放口径
- [ ] `PRD` 中的用户故事和验收标准与已冻结能力一致
- [ ] `PRD` 没有把待补能力写成既有能力

### 4.2 SPEC

- [ ] `SPEC` 的路径、schema、返回码与权威源一致
- [ ] `SPEC` 没有出现正文未解释的字段处理差异
- [ ] `SPEC` 对待补能力的描述不超过当前 Core 事实
- [ ] 若为对象存储，`SPEC` 已明确“当前只承接对象元数据”，没有把上传/下载写成既有冻结能力

### 4.3 主维护源

- [ ] 主维护源是当前模块唯一事实源
- [ ] 主维护源比 `PRD / SPEC` 更稳定、更收口
- [ ] 主维护源已说明字段展示规则、字段口径、待补边界、删除契约边界

### 4.4 HTML 摘要

- [ ] HTML 只保留摘要、边界、入口
- [ ] HTML 没有重新引入旧路径、旧错误码、旧边界
- [ ] HTML 与主维护源口径一致
- [ ] HTML 没有把对象元数据创建写成“文件上传”或其他未冻结动作

## 5. 第三轮：反自造契约检查

以下任一命中都视为阻塞项：

- [ ] 没有自造 schema 名称
- [ ] 没有自造字段名
- [ ] 没有自造状态枚举
- [ ] 没有自造错误码
- [ ] 没有自造动作路径
- [ ] 没有把未来可能设计写成当前冻结事实

## 6. 第四轮：页面能力降噪检查

- [ ] 页面能力没有写得超过现有 schema/接口支撑范围
- [ ] 没有把“风险提示”写成“已有详情展示能力”
- [ ] 没有把“跳转入口”写成“已有聚合数据能力”
- [ ] 没有把“请求控制字段”写成“用户展示字段”
- [ ] 没有把“开发态辅助字段”误写成默认首屏展示字段
- [ ] 若为对象存储，没有把“对象元数据创建”误写成“文件上传”
- [ ] 若为对象存储，没有把 `bucket` 字段误写成独立桶资源列表/详情能力

## 7. 第五轮：最终答复范围检查

- [ ] 最终答复已明确说明本次结论覆盖“当前模块”还是“全量可用模块”
- [ ] 若只复审了当前模块，没有使用“现在写的内容都符合”“全部都通过”这类会让用户误解为全量串审的表述
- [ ] 若输出全量结论，确实已经完成了全量重新串审

## 8. 强制阻塞项

命中以下任一项，必须先修复，禁止交付“已达到 Core 层要求”结论：

- [ ] 文档中存在权威源里没有的 schema 名称
- [ ] 文档中存在权威源里没有的字段、状态或错误码
- [ ] 文档中把待补能力写成正式接口表
- [ ] 文档中把请求字段写成用户展示字段
- [ ] 文档中把无字段支撑的页面能力写成既有能力
- [ ] `PRD / SPEC / 主维护源 / HTML` 中任意一份仍保留旧路径或旧边界
- [ ] 对象存储文档把“对象元数据创建”写成“文件上传”或“内容上传已完成”
- [ ] 对象存储文档把 `bucket` 字段写成独立桶资源已冻结能力
- [ ] 最终答复未明确本次确认范围，却直接给出“都已符合”类表述

## 9. 最终结论模板

复审结论只能二选一：

### 8.1 通过

- 已达到 `Core` 层要求
- 路径、schema、返回码、租户边界、待补能力边界均已对齐
- `PRD / SPEC / 主维护源 / HTML` 已一致
- 当前不存在阻塞项
- 已明确本次结论的确认范围

### 8.2 不通过

- 尚未达到 `Core` 层要求
- 阻塞项如下：
  - [阻塞项 1]
  - [阻塞项 2]
- 必须先完成修正，再重新执行本清单
